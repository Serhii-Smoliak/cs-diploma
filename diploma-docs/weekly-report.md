### **1\. Опрацювання початкової концепції проєкту**

CyberTactics — це інженерний cybersecurity-проєкт у форматі навчальної гейміфікованої вебплатформи для вивчення логіки кібератак, технік MITRE ATT&CK та етапів Cyber Kill Chain.

Основна проблема, яку вирішує проєкт: новачкам складно почати вивчати кібербезпеку через високий поріг входу. Платформи на кшталт Hack The Box або TryHackMe часто потребують попередніх знань: Linux, мережі, термінал, VPN, базова exploit-логіка, web-security. Через це користувач без бази швидко стикається не з навчанням, а з технічним бар’єром.

CyberTactics має вирішити цю проблему через просту браузерну симуляцію атак, де користувач поступово проходить місії, виконує короткі завдання та отримує пояснення:

* що саме відбувається;
* навіщо використовується конкретна техніка;
* який результат вона дає;
* як вона пов’язана з наступним етапом атаки;
* як це співвідноситься з MITRE ATT&CK.

Проєкт орієнтований на школярів, студентів, junior pentesters, розробників, QA, DevOps/DevSecOps та інших спеціалістів суміжних IT-напрямів, які хочуть зрозуміти offensive security з нуля.

Практична цінність проєкту полягає в тому, що він не просто дає набір технічних задач, а допомагає зібрати знання в єдину послідовну систему через місії, сценарії атак, MITRE ATT&CK, гейміфікацію та поетапне пояснення логіки атак.

Поточна концепція CyberTactics уже описана як освітня гейміфікована платформа для вивчення кібербезпеки через симуляцію сценаріїв атак, де опорою для контенту є MITRE ATT&CK.

### **2\. Попередній аналіз можливих підходів**

Було розглянуто кілька підходів до навчання кібербезпеці:

| **Підхід**                     | **Переваги**                                        | **Недоліки**                                                   |
|--------------------------------|-----------------------------------------------------|----------------------------------------------------------------|
| Класичні курси                 | Добре пояснюють теорію                              | Мало практики, слабкий зв’язок із реальними атаками            |
| CTF / Hack The Box / TryHackMe | Дають практику                                      | Складні для повних новачків                                    |
| Лабораторії з VM               | Реалістичні                                         | Потребують налаштування середовища                             |
| Тести awareness                | Прості                                              | Часто нудні й поверхневі                                       |
| CyberTactics                   | Простий старт, поетапна логіка, MITRE, гейміфікація | Потребує якісного навчального контенту та контролю етичних меж |

Обраний підхід: браузерна навчальна платформа, яка не запускає реальний шкідливий код, а симулює логіку атаки та перевіряє розуміння користувача через завдання. Для MVP використовується перевірка логіки відповіді, а не виконання небезпечного коду.

### **3\. Цілі проєкту**

Основні цілі проєкту:

1. Розробити MVP навчальної платформи з кібербезпеки.
2. Знизити поріг входу для користувачів без досвіду.
3. Пояснювати offensive security через прості поетапні завдання.
4. Використати MITRE ATT&CK як основу для структурування навчального контенту.
5. Реалізувати місії, рівні, прогрес користувача та гейміфікацію.
6. Дати користувачу розуміння не лише “що зробити”, а й “навіщо це робиться”.
7. Створити безпечне навчальне середовище без виконання реального шкідливого коду.
8. Підготувати основу для подальшого розширення контенту, сценаріїв і defensive-пояснень.

### **4\. Опрацювання первинної версії MVP**

На першому тижні було підготовлено та частково реалізовано початковий MVP CyberTactics.

**Реалізовано**:

| **Напрям**     | **Поточний результат**                                             |
|----------------|--------------------------------------------------------------------|
| Frontend       | React + Vite, сторінки місій, завдань, ігровий екран, Skill Matrix |
| Backend        | Express API, авторизація JWT, робота з місіями, рівнями, прогресом |
| База даних     | PostgreSQL + Prisma ORM                                            |
| MITRE ATT&CK   | Синхронізація STIX-даних, збереження технік у БД                   |
| Гейміфікація   | XP, ранги, Stealth, прогрес користувача                            |
| Завдання       | Code editor, tactical choice, phishing constructor                 |
| Локалізація    | i18next + переклади з БД                                           |
| Інфраструктура | Docker Compose для PostgreSQL, Dockerfile для backend              |

У документації MVP зафіксовано, що вже реалізовано реєстрацію/вхід, CRUD місій і рівнів, відправку відповідей, прогрес, XP, Stealth, MITRE sync, Skill Matrix, переклади та базову локалізацію.

Архітектура побудована як frontend → backend API → PostgreSQL. Frontend працює через React + Vite, backend — через Express, middleware, services та Prisma.

### **5\. Початкова модель загроз**

Для першого етапу сформовано початкову модель загроз саме для навчальної cybersecurity-платформи.

#### **Основні активи**

* акаунти користувачів;
* JWT-токени;
* прогрес проходження;
* XP, ранги, Stealth;
* навчальний контент;
* місії та рівні;
* MITRE ATT&CK дані;
* переклади;
* адміністративні endpointʼи;
* механізм перевірки відповідей.

#### **Довірені компоненти**

* backend-сервер;
* база даних PostgreSQL;
* серверна логіка перевірки відповідей;
* офіційне джерело MITRE CTI;
* контрольовані JSON-сценарії рівнів.

#### **Недовірені компоненти**

* користувацьке введення;
* відповіді на завдання;
* frontend-стан у браузері;
* localStorage;
* будь-які API-запити з клієнта;
* потенційно змінюваний навчальний контент;
* незахищені адміністративні endpointʼи.

#### **Початковий STRIDE-аналіз**

| STRIDE                 | Потенційний ризик              | Приклад                                          |
|------------------------|--------------------------------|--------------------------------------------------|
| Spoofing               | Підміна користувача            | Використання чужого JWT                          |
| Tampering              | Зміна прогресу або результатів | Спроба підмінити XP, Stealth або completed level |
| Repudiation            | Заперечення виконаних дій      | Користувач заперечує проходження рівня           |
| Information Disclosure | Витік даних                    | Email, прогрес, службові API, навчальний контент |
| Denial of Service      | Перевантаження системи         | Масові submit-запити або MITRE sync              |
| Elevation of Privilege | Доступ до admin-функцій        | Виклик MITRE sync або translations API без прав  |

Поточні важливі ризики: MITRE sync і translations API поки потребують admin-захисту, а AST-валідація коду реалізована спрощено.

**Зручність ігрового інтерфейсу**

* Після успішного виконання можна натиснути «Пройти заново» і повторити рівень.

**Механіка Stealth**

Stealth імітує ризик викриття операції. Помилки зменшують рівень прихованості. Якщо Stealth падає до нуля, гравець не може надіслати відповідь, доки не відновить ресурс.

Подія — ефект:

* Старт — 100%.
* Невірна відповідь — мінус 5%, значення налаштовується.
* Час — поступове відновлення.
* Stealth дорівнює 0 — блокування надсилання відповіді та вікно відновлення.

Варіанти відновлення в демо без реальної оплати:

* купити маскування — швидке відновлення до 50%;
* платний тариф — заготовка для майбутньої реалізації;
* зачекати — часткове відновлення після паузи.

Параметри інтервалу, штрафу та відновлення задаються в налаштуваннях сервера без зміни коду.

На екрані додано індикатор у шапці: зелений, жовтий або червоний стан. При спробі відповісти з нульовим Stealth відкривається модальне вікно відновлення.

### **Delta threat model**

Після додавання ролей користувачів, admin-захисту та ownership-перевірок модель загроз змінилась. 

Було:

* критичні endpointʼи могли бути потенційно доступні без достатнього контролю ролей;
* progress/stats потребували явної перевірки власника;
* MITRE sync і translations API були окремими ризиковими точками.

Стало:

* admin endpointʼи обмежуються RBAC;
* MITRE sync доступний лише адміністратору;
* translations API захищений ролями;
* progress/stats перевіряються через userId з JWT;
* XP/Stealth не приймаються напряму з клієнта, а мають обчислюватися на backend.

### Детальний аналіз

#### 1\. Path Traversal на translations API

OWASP ZAP виявив потенційний ризик Path Traversal для endpointʼа:

GET /api/translations/namespaces?namespaces=namespaces

Проблема повʼязана з параметром namespaces, який потенційно може бути використаний для спроби доступу до небажаних ресурсів або обходу очікуваної логіки API.

Категорія:

* OWASP 2021 A01 — Broken Access Control
* CWE-22 — Improper Limitation of a Pathname to a Restricted Directory

Оцінка:

* Ризик: High
* Впевненість: Low

Висновок: це потребує ручної перевірки. Навіть якщо фактичного доступу до файлової системи немає, параметр namespaces треба обмежити whitelist-логікою та серверною валідацією.

План виправлення:

* дозволити тільки відомі namespace-значення;
* заборонити ../, /, \\, encoded path payloads;
* додати DTO/schema validation;
* не використовувати user input напряму для побудови шляхів або динамічних ключів;

#### 2\. Відсутній заголовок X-Content-Type-Options

ZAP виявив, що відповіді API не містять заголовок:

X-Content-Type-Options: nosniff

Цей заголовок зменшує ризик MIME-sniffing атак у браузері.

Категорія:

* OWASP 2021 A05 — Security Misconfiguration
* CWE-693 — Protection Mechanism Failure

Оцінка:

* Ризик: Low
* Впевненість: Medium

План виправлення:

* підключити helmet в Express;
* явно встановити X-Content-Type-Options: nosniff;
* перевірити повторним ZAP scan.

#### 3\. Розкриття технології через X-Powered-By

Backend повертає заголовок:

X-Powered-By: Express

Це не є критичною вразливістю, але допомагає fingerprintingʼу технологічного стеку.

Категорія:

* OWASP 2021 A05 — Security Misconfiguration
* CWE-497 — Exposure of Sensitive System Information

Оцінка:

* Ризик: Low
* Впевненість: Medium

План виправлення:

* вимкнути x-powered-by в Express;
* використовувати helmet.

#### 4\. Suspicious comments у відповіді API

У відповіді endpointʼа місій знайдено коментар:

TODO: Remove before prod

Це може розкривати внутрішню інформацію про стан розробки або незавершені частини функціоналу.

Категорія:

* CWE-615 — Inclusion of Sensitive Information in Source Code Comments

Оцінка:

* Ризик: Informational
* Впевненість: Medium

План виправлення:

* прибрати debug/TODO-коментарі з response;
* не повертати службову інформацію у production-відповідях;
* розділити dev/test/prod seed data.

#### 5\. Authentication request identified

ZAP визначив endpoint:

POST /api/auth/login

як запит автентифікації, оскільки він містить поля:

* email
* password

Це informational alert, а не вразливість.

Висновок: endpoint login коректно ідентифікований ZAP як authentication flow. Додатково потрібно перевірити rate limiting, generic error messages, password policy та захист від brute force.

#### 6\. Session management response identified

ZAP визначив, що відповідь login endpointʼа містить session/token marker:

token

Це informational alert, а не самостійна вразливість.

Висновок: потрібно окремо перевірити:

* expiration JWT;
* алгоритм підпису;
* відсутність sensitive data в payload;
* поведінку після logout;
* зберігання токена на frontend;
* захист від XSS.

### Звʼязок результатів OWASP ZAP зі STRIDE

Результати первинного OWASP ZAP scan було зіставлено з попередньою STRIDE-моделлю CyberTactics. Це дозволяє не просто зафіксувати alerts, а повʼязати їх із конкретними ризиками для платформи та визначити план подальших виправлень.

| **STRIDE**             | **Виявлений ризик**                       | **Як проявляється у CyberTactics**                                                             | **Контроль / план виправлення**                                                               |
|------------------------|-------------------------------------------|------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------|
| Spoofing               | Потенційне зловживання auth/session flow  | ZAP визначив /api/auth/login як authentication endpoint та виявив token-based session response | Перевірити JWT expiration, signature validation, generic auth errors, rate limiting для login |
| Tampering              | Маніпуляція параметром namespaces         | ZAP виявив потенційний Path Traversal у GET /api/translations/namespaces?namespaces=namespaces | Додати whitelist allowed namespaces, schema validation, заборону ../, /, \\, encoded payloads |
| Repudiation            | Недостатня простежуваність критичних дій  | Для admin-операцій, MITRE sync, translations changes та submit actions потрібна історія дій    | Додати audit logging для admin-запитів, sync, змін перекладів і submit results                |
| Information Disclosure | Розкриття службової інформації            | Виявлено X-Powered-By: Express, TODO-коментар у response, auth/session markers                 | Вимкнути X-Powered-By, прибрати debug/TODO-коментарі, мінімізувати response body              |
| Denial of Service      | Потенційне перевантаження API             | Login, submit, translations та MITRE sync можуть бути викликані багато разів                   | Додати rate limiting, cache для translations, обмеження sync endpoint                         |
| Elevation of Privilege | Доступ до критичних API без належної ролі | Translations API та MITRE sync є потенційними admin-only зонами                                | Додати RBAC middleware, admin guard, ownership checks                                         |

* **Dependabot** — `.github/dependabot.yml`: щотижневі оновлення для shared, server, client і GitHub Actions.
* **Snyk** — matrix job для server і client; поріг `severity-threshold=high`; secret `SNYK_TOKEN`.

#### 4. Container scan

* Job `container-scan` збирає Docker image backend (`server/Dockerfile`) і сканує його Trivy (`aquasecurity/trivy-action@0.35.0`).
* Налаштовано `scanners: vuln`, `severity: CRITICAL,HIGH`, `ignore-unfixed: true`.
* Для зменшення false positives додано `skip-dirs` для системних npm-шляхів у базовому Node image.

#### 5. Docker і Railway

* Оновлено `server/Dockerfile` для monorepo: збірка shared + server, `npm ci`, `npm prune --omit=dev`, `apk upgrade` для системних пакетів.
* Додано `railway.toml` у корені репо для backend: build shared+server, start `node dist/index.js`.
* Міграція з yarn на npm: `package-lock.json` у server/client/shared, оновлено CI, Dockerfile, Railway і документацію.
* Backend (`cs-diploma-be`) і frontend (`cs-diploma-fe`) на Railway у статусі Online після merge в `main`.

#### 6. Branch protection і deploy flow

Налаштовано deploy flow для MVP:

```
Pull Request → GitHub Actions (усі jobs green) → merge в main → Railway auto-deploy
```

На гілці `main` увімкнено branch protection:

* require pull request before merging;
* require status checks: `Build & lint`, `Dependency audit`, `SonarCloud SAST`, `Snyk dependencies (server)`, `Snyk dependencies (client)`, `Container scan (Trivy)`;
* заборона direct push у `main`.

PR #16 змін CI/CD pipeline змержено в `main` після проходження всіх checks.

#### 7. Документація і security gates

* Створено `diploma-docs/ci-cd-pipeline.md` — схема pipeline, опис jobs, security gates, secrets, deploy flow.
* Оновлено `README.md`, `Security-requirements.md`, `DOCUMENTATION.md`.

Security gates блокують merge/deploy, якщо:

* build, lint або typecheck не пройшли;
* npm audit знайшов critical CVE;
* Sonar Quality Gate failed;
* Snyk знайшов high/critical вразливості в залежностях;
* Trivy знайшов CRITICAL/HIGH CVE в Docker image;
* відсутні потрібні GitHub Secrets.

#### 8. Додаткові виправлення для стабільного CI

Під час налаштування pipeline були усунені блокери збірки та деплою:
